## 0. Default policy

**Anything not explicitly listed as "Enabled" in §1 is disabled by default for the pilot**, including any module or flag not named anywhere in this document — an unverified/unlisted module is never enabled by omission. This mirrors [../../program/LAUNCH_GATES.md](../../program/LAUNCH_GATES.md) §2.G's activation-progression rule: `code merged → migration applied → feature disabled (default) → internal activation → pilot activation (explicit, monitored)`. Nothing in this matrix moves any feature past "disabled" on its own — enabling a module for a specific pilot clinic is a separate, recorded action taken during that clinic's onboarding (per [PILOT_CUSTOMER_ONBOARDING_CHECKLIST.md](PILOT_CUSTOMER_ONBOARDING_CHECKLIST.md) §3 step 6), not an automatic consequence of this document existing.

## 1. Allowed initial modules

| Module | Status | Prerequisite evidence before enabling for a clinic |
|---|---|---|
| Patients | **Enabled** | Clinic passed [PILOT_CLINIC_ACCEPTANCE_CRITERIA.md](PILOT_CLINIC_ACCEPTANCE_CRITERIA.md) §4; tenant/user/legal-profile creation complete ([PILOT_CUSTOMER_ONBOARDING_CHECKLIST.md](PILOT_CUSTOMER_ONBOARDING_CHECKLIST.md) §2 checks 5-7); cross-tenant negative tests for patient records passing at the deployed commit ([../../program/LAUNCH_GATES.md](../../program/LAUNCH_GATES.md) §2.C) |
| Appointments | **Enabled** | Same as Patients; appointment-scheduling tests passing at the deployed commit |
| Payments | **Enabled** | Same as Patients; payment/ledger tests passing at the deployed commit; clinic has confirmed its own currency/billing configuration (`currency` set at tenant creation) |
| Basic communication (respecting consent) | **Enabled**, conditional | WhatsApp/Meta configuration complete for this clinic ([PILOT_CUSTOMER_ONBOARDING_CHECKLIST.md](PILOT_CUSTOMER_ONBOARDING_CHECKLIST.md) §2 check 9); KVKK-HIGH-007 consent-enforcement flags' pilot value explicitly recorded (audit-mode is acceptable if explicitly chosen, per [../../program/LAUNCH_GATES.md](../../program/LAUNCH_GATES.md) §2.G); a clinic without completed WhatsApp/Meta configuration stays on this row as **disabled** until it is complete — the module itself is allowed, per-clinic activation is still conditional |
| Contact requests | **Enabled** | Cross-tenant negative tests for the contact-request path passing at the deployed commit |
| Limited reporting | **Enabled**, scoped | Only reports that do not constitute a bulk export (see §2) — e.g. clinic's own patient/appointment/payment summaries, not a full-database or cross-clinic report |

"Enabled" above means "allowed to be turned on for a pilot clinic once its own prerequisite evidence is met" — it is not a statement that any of these are currently turned on for any clinic, since no clinic is yet onboarded as of this baseline.

## 2. Restricted modules/flows

| Module/flow | Status | Prerequisite evidence to change status |
|---|---|---|
| HIGH-008 legacy-consent-correction workflow | **Disabled — manual approval required, and controlled activation is currently `CONTROLLED_ACTIVATION_NOT_AUTHORIZED`** ([../../program/RISK_REGISTER.md](../../program/RISK_REGISTER.md) R-061, gap 9) | The `privacy.legacyConsentCorrection.runtimeEnabled` `PlatformSetting` gate ([../../program/KVKK_HIGH008_FREEZE_BOUNDARY.md](../../program/KVKK_HIGH008_FREEZE_BOUNDARY.md)) stays `false`/absent for every pilot clinic for the duration of this package's validity. Enabling it requires its own separate, explicit controlled-activation decision (named approver, maintenance window, rollback/cutback plan, monitoring plan, pre-agreed evidence-capture scope) per R-061 — this matrix does not authorize that decision and no pilot clinic's onboarding may be made contingent on it. |
| Uncontrolled multi-branch use | **Disabled** (structurally excluded, see [PILOT_CLINIC_ACCEPTANCE_CRITERIA.md](PILOT_CLINIC_ACCEPTANCE_CRITERIA.md) §2 item 1) | A clinic requiring multiple branches is not eligible for this pilot at all under current acceptance criteria; would require its own tenant-isolation re-assessment before consideration, out of scope here |
| Bulk exports (`clinicBulkExport`) | **Disabled — manual approval required per export** | This feature exists in code (`server/src/routes/clinicBulkExport.ts`, password-protected packaging, worker/cleanup jobs). It is not blanket-enabled for the pilot because it directly exercises the same storage/backup/entitlement gaps ([../../program/RISK_REGISTER.md](../../program/RISK_REGISTER.md) R-029/R-030, R-025 entitlement-enforcement `UNVERIFIED`) that make the pilot's data-volume boundedness (§2 item 2 of [PILOT_CLINIC_ACCEPTANCE_CRITERIA.md](PILOT_CLINIC_ACCEPTANCE_CRITERIA.md)) load-bearing. A specific export may be manually approved by the decision owner on a per-request basis, with the requesting clinic and purpose recorded, rather than exposed as a self-service feature. |
| Bulk messaging/campaigns | **Disabled / not applicable** | As of this baseline, no bulk-messaging or campaign-send feature exists in this codebase — `'campaign'` is only a consent-purpose value in the communication-consent taxonomy (`server/src/services/communicationConsent/taxonomy.ts`), not a sending capability. This row exists to record the restriction explicitly per this task's requirement; there is currently nothing to disable beyond §1's single-message "basic communication," which is itself consent-gated. If a bulk-send capability is added to the codebase in the future, it inherits this row's "disabled — manual approval required" status by default, per §0. |
| DICOM/imaging | **Disabled / not applicable to pilot scope** | No DICOM viewer or PACS integration exists in this codebase; the narrow imaging-bridge file-intake contract that does exist ([../../47-imaging-bridge-contract.md](../../47-imaging-bridge-contract.md), [../../48-imaging-bridge-agent.md](../../48-imaging-bridge-agent.md)) is explicitly gated to a later program phase/release gate (G4) per [../../program/ARCHITECTURE_DECISIONS.md](../../program/ARCHITECTURE_DECISIONS.md) ADR-011 and is out of scope for this pilot regardless of any individual clinic's request. |
| Official health integrations | **Disabled / not applicable to pilot scope** | No official health-registry integration exists in this codebase as of this baseline; out of scope for this pilot. |
| Medical AI decision support | **Disabled / not applicable to pilot scope** | No medical AI decision-support feature exists in this codebase as of this baseline; out of scope for this pilot. |
| Unverified modules (anything not named in §1) | **Disabled by default** | Per §0 — any module not explicitly listed in §1 requires its own future addition to this matrix, with its own prerequisite-evidence row, before it may be enabled for any pilot clinic. |

## 3. Explicit distinction: pilot-ready, general-launch-ready, production-verified

These three states are **not the same thing** and are never collapsed into one another, consistent with the non-collapse principle in [../../program/LAUNCH_GATES.md](../../program/LAUNCH_GATES.md) §0.

| State | Meaning | What satisfies it |
|---|---|---|
| **Pilot-ready** | A module has met the prerequisite evidence in §1/§2 of *this* document, sufficient to be enabled for a small, named, actively monitored pilot cohort of at most 3 clinics under managed onboarding. | The specific prerequisite-evidence column for that module in §1, plus the clinic-level go/no-go gate in [PILOT_CLINIC_ACCEPTANCE_CRITERIA.md](PILOT_CLINIC_ACCEPTANCE_CRITERIA.md) §4. |
| **General-launch-ready** | A module (and the platform overall) has met [../../program/LAUNCH_GATES.md](../../program/LAUNCH_GATES.md) §3 (G2)'s requirements — self-service, uncontrolled-count onboarding, mandatory (not merely accepted-temporary) offsite backup/PITR/restore, CI coverage, monitoring/alerting, automated rollback, and independent security review of tenant isolation. | G2 evaluation per [../../program/LAUNCH_GATES.md](../../program/LAUNCH_GATES.md) §3 and §7 — **not evaluated by this pilot package, and not implied by any status in §1/§2 above.** Pilot-ready never implies general-launch-ready. |
| **Production-verified** | A specific claim about production behavior has been checked against production itself (a live command, query, or test execution against the real production system) and the result recorded — distinct from "code merged," "tests passed in a disposable/local environment," or "deployed," per [../../program/LAUNCH_GATES.md](../../program/LAUNCH_GATES.md) §0's non-collapse table. | A dated, evidence-linked production check. As of this baseline, most of §1's modules have **not** had this pilot-specific production-verification performed — [PILOT_FIRST_WEEK_MONITORING_PLAN.md](PILOT_FIRST_WEEK_MONITORING_PLAN.md)'s first-day smoke checklist is the mechanism that produces this evidence, once actually executed against a real onboarded clinic. |

A module appearing "Enabled" in §1 is a statement that it is **allowed to become pilot-ready** once its own prerequisite evidence is met for a specific clinic — it is neither general-launch-ready nor production-verified by virtue of appearing in this table. Do not cite this matrix as proof that any module has passed G1, G2, or a live production check; cite the actual dated evidence produced by following [PILOT_CUSTOMER_ONBOARDING_CHECKLIST.md](PILOT_CUSTOMER_ONBOARDING_CHECKLIST.md) and [PILOT_FIRST_WEEK_MONITORING_PLAN.md](PILOT_FIRST_WEEK_MONITORING_PLAN.md) instead.

## 4. Maintenance of this matrix

Any change to a module's status (enabling a previously restricted module, disabling a previously enabled one) is a decision-owner action, recorded with date, reason, and evidence, per [../../program/LAUNCH_GATES.md](../../program/LAUNCH_GATES.md) §0's "no agent may self-assign" rule. This file itself is not a shared program tracker file within the meaning of this task's constraints — it is a pilot-package-local operational document — but it should still be kept current, not silently allowed to drift from actual production configuration (see [PILOT_FIRST_WEEK_MONITORING_PLAN.md](PILOT_FIRST_WEEK_MONITORING_PLAN.md) §3 item 9's daily drift check).
