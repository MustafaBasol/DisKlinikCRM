## 5. Exact event that moves the freeze boundary

The freeze boundary for the areas in §2 marked MUTABLE moves only when one of the following occurs (all require external evidence — an agent's own "done" claim never qualifies, per tracker §2.2/§2.3):

1. The active KVKK-HIGH-007 continuation is committed to a branch and a PR is opened against `main` (moves status from `UNVERIFIED_ACTIVE_WORK` to `PR_OPEN`, evidenced by `gh pr view <n>`). **Satisfied as of this correction pass** — [PR #175](https://github.com/MustafaBasol/DisKlinikCRM/pull/175), `OPEN`, opened `2026-07-19T13:19:58Z`. This condition alone only changes the *evidence-availability* status (content is now theoretically reviewable on GitHub); it does not itself unfreeze anything and content remains uninspected by this task.
2. That PR is merged (`MERGED`, evidenced by `gh pr view <n> --json state,mergedAt,mergeCommit`) — this is the primary event that unfreezes areas 1-11 and 20 in §2, since it fixes the schema/migration/service state those areas depend on. **Satisfied as of this F0-008 correction pass (2026-07-19)** — `git log --oneline -1 origin/main` shows `1da9586 KVKK-HIGH-007 follow-up: legacy/central consent reconciliation, audit-mode readiness, UX hardening (#175)`; `git diff --stat 7cf7a82 1da9586` inspected directly, confirming the schema/migration/service changes described in §2's rows. §2 rows 1-11 and 20 are updated accordingly above.
3. The merged migration is applied in production and confirmed via read-only production evidence (`VERIFIED_PRODUCTION_OBSERVED`, same pattern as F0-002 Stage B) — required before any claim of "deployed." **SATISFIED as of 2026-07-20** (post-deployment production-evidence reconciliation pass) — production `HEAD` `87b7dcd9bfe5d27956e32e8b61590667e36bac86`, 64/64 migrations present including `20260719120821_kvkk_high007_consent_reconciliation`, `prisma migrate status` reports the schema up to date. This satisfies migration-*application* only; it is not, by itself, a "deployed" or "production ready" claim for the surrounding feature behavior — see condition 4.
4. A rollback/tenant-impact verification is independently evidenced (not merely asserted) — required before any claim of "production ready" for schema-touching KVKK work, per the task's mandatory rule that migration validation/rollback/tenant-isolation impact/consent-conflict-resolution behavior/production readiness all remain `UNVERIFIED`/`NOT ESTABLISHED` until then. **Not satisfied.**
5. External review/user acceptance explicitly declares the KVKK baseline "stable" (the qualitative trigger named throughout program docs as "KVKK taban çizgisinin dış teyidi"). **Not satisfied.**

Because conditions 4-5 remain unsatisfied, this document does **not** claim the freeze boundary is fully lifted — as of 2026-07-20, conditions 2 (merged) and 3 (migration confirmed applied in production) are both met, which retires the *concurrent-writer* risk in §2 rows 1-11/20 (STABLE merged baseline) and confirms the schema change is live, without authorizing production activation of enforcement/reconciliation, RLS/tenant-extension implementation, or any §3 default-freeze item, and without independently verifying rollback safety or tenant-isolation impact. Areas 12-19 in §2 are governed by the pre-existing, broader program-wide freeze (tracker §8) and move on **that** freeze's own conditions (F4/F5/F6/F9/F10/F11 entry gates, KVKK baseline stabilization generally) — they are not specifically gated by the KVKK-HIGH-007 continuation alone, though a stabilized KVKK baseline (condition 5, still unsatisfied) is a shared precondition. The §3 default freeze rules (RLS rollout, Prisma tenant-extension rollout, broad schema refactoring, queue/outbox implementation, message-delivery refactor, broad recall refactor) are likewise gated on condition 5, not conditions 2-3, and remain in force.
