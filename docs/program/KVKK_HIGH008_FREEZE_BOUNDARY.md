### 7.1 Correction (2026-07-20, external review pass) — rollback/cutback claim was wrong as originally stated

The first pass of this section (and the accompanying delivery report) stated that an older application version simply "ignores" the `PlatformSetting` and that this is "a harmless no-op." **That framing is incorrect from a control perspective and is corrected here:**

- **Schema backward compatibility: yes.** The `PlatformSetting` table is unchanged and additive; no migration was made or is needed.
- **Application compatibility with the additive schema: yes.** Any application version, old or new, can read/write `PlatformSetting` rows without schema error.
- **Runtime-control compatibility when rolling back to a pre-gate application commit: no.** An application version that predates this task's gate check (i.e. any commit before the one adding the `isLegacyConsentCorrectionRuntimeEnabled()` check to `communicationPreferences.ts`) does not read `privacy.legacyConsentCorrection.runtimeEnabled` at all. Such a version re-exposes the legacy-correction mutation route unconditionally to OWNER/ORG_ADMIN/CLINIC_MANAGER regardless of what the setting's stored value is. A blind rollback to that commit — for example as an incident-response "just revert the deploy" reflex — silently defeats the kill switch this task exists to provide, at the exact moment (an incident) the kill switch is most likely to be needed.

**Required cutback statement:** Do not use a blind rollback to a pre-gate application commit as the feature-disable mechanism. A safe cutback must retain runtime-gate enforcement, use a compatible prior commit containing the gate, add a temporary route-level block, or forward-fix.

In practice, once this branch merges, the safe ways to disable the workflow are, in order of preference: (1) PATCH the setting to `false` via the platform-admin route (no deploy needed, effective on the next request — see §7's caching note); (2) if a code rollback is independently required for an unrelated reason, roll back only to a commit at or after this gate's merge commit, never before it; (3) if neither is available, apply a temporary route-level block (e.g. a hard `return res.status(503)` at the top of the mutation route) as an emergency stopgap, not a rollback to older code.

### 7.2 Second correction (2026-07-20, same day) — `SecuritySignalEvent` was the wrong durable-audit store

§7.1's fix left one part of the first-pass implementation itself flawed: the durable audit record for the admin enable/disable toggle (§7 above) was implemented by reusing `SecuritySignalEvent` (existing KVKK-CRIT-003 security-detection infrastructure) via `recordSecuritySignal()`, on the reasoning that its `organizationId` is nullable and it already has an `actorPlatformAdminId` field. **A second, independent external review found this reasoning insufficient:**

- `SecuritySignalEvent` is a security-detection telemetry store, aggregated by `securityDetectionRules.ts` into incident escalation (`securityIncidentService.ts`). A platform-admin configuration change (enabling/disabling a KVKK kill switch) is not a security-detection signal — it is routine administrative action, attributable and auditable, but conceptually a different kind of event.
- Reusing the same table mixes two distinct domains (config-change audit vs. security-incident detection), which creates retention/reporting ambiguity (a compliance audit query for "who changed this setting and when" would need to filter out unrelated security telemetry, or vice versa) and risks a future detection rule broadening its aggregation criteria and unintentionally treating routine admin changes as security signals.
- That no *current* rule in `securityDetectionRules.ts` reads this specific `signalType`/`ruleKey` does not make the underlying data-model choice correct — it only means today's rules happen not to be affected.

**Correction:** a dedicated, additive `PlatformAdminAuditEvent` Prisma model was added (migration `20260720180000_add_platform_admin_audit_event` — `id`, `actorPlatformAdminId` with a real FK to `PlatformAdmin` (`onDelete: SetNull`), `action`, `resourceType`, `resourceKey`, `previousValue`, `newValue`, `outcome`, optional `safeMetadata`, `createdAt`; indexed on `(actorPlatformAdminId, createdAt)` and `(action, createdAt)`; create-only, no update/delete path), plus a narrow service (`server/src/services/platformAdminAudit.ts`, `writePlatformAdminAuditEventInTx()`, mirroring the existing `writeAuditLogInTx()` pattern in `utils/auditLog.ts` — accepts only a `Prisma.TransactionClient`, no swallow-all-errors fallback; see §7.3 for the compile-time-enforcement correction to this function's type signature). The PATCH route now performs the `PlatformSetting` upsert and the audit insert inside a single `prisma.$transaction(...)` call, so a failed audit insert rolls back the setting change too. This was proven, not just asserted: a test forces a real foreign-key-constraint violation (a non-existent `actorPlatformAdminId`) and confirms both that the whole request rejects and that the `PlatformSetting` row is left unchanged. The `SecuritySignalEvent` write this feature had added is removed entirely; a test confirms zero `SecuritySignalEvent` rows are created for this action.

No production system was touched by this second correction, and R-061 remains `OPEN` — this is a repository-level correctness fix to an unmerged branch, not a deployment or activation event.

### 7.3 Third correction (2026-07-20, same day) — previous-value read was outside the atomic boundary; TransactionClient requirement was documented, not enforced

An independent, read-only review of the branch (after §7.2) found two remaining gaps, both narrow and both corrected here without expanding this feature's scope:

- **Previous-value read was not part of the atomic write.** The PATCH route (`platformAdmin.ts`) read `previousEnabled` via `isLegacyConsentCorrectionRuntimeEnabled()` *before* opening `prisma.$transaction(...)`, on its own connection with no lock. Under concurrent admin PATCH requests, a second request's pre-transaction read could observe a stale value from before the first request's commit, producing an audit row whose `previousValue` does not match the actual committed write order — a real gap in the "durable, sequentially correct audit chain" goal §7.2 set out to establish, not merely a cosmetic timing issue.
- **`writePlatformAdminAuditEventInTx()`'s "REQUIRES a Prisma.TransactionClient" claim (§7.2) was a documented convention, not a compile-time guarantee.** Its actual signature accepted `Pick<PrismaClient, 'platformAdminAuditEvent'> | Prisma.TransactionClient` — a union the global `prisma` singleton also structurally satisfies, so TypeScript would not have stopped a future caller from passing `prisma` directly and silently defeating the atomicity guarantee. The one call site in this branch always passed a real `tx`, so this was a latent-misuse risk, not a live bug.

**Correction:**
- The previous-value read now happens *inside* the same `prisma.$transaction(...)` that performs the `PlatformSetting` upsert and the `PlatformAdminAuditEvent` insert, immediately before the upsert. `getPlatformSetting()` and `isLegacyConsentCorrectionRuntimeEnabled()` both gained an optional transaction-client parameter (mirroring `setPlatformSetting()`'s existing optional-client pattern — no new settings framework), defaulting to the global `prisma` singleton so every other call site (the GET policy route, the mutation-gate check in `communicationPreferences.ts`) is unaffected.
- Because `PlatformSetting` has no version/lock column and a never-toggled key has no existing row for `SELECT ... FOR UPDATE` to lock, the transaction first acquires a transaction-scoped Postgres advisory lock keyed to this one setting (`pg_advisory_xact_lock(hashtext(key))`) before reading. This serializes concurrent PATCH transactions on this single key — a second transaction blocks until the first commits and releases the lock automatically, then reads the just-committed value — without taking any table- or row-level lock and without affecting any other `PlatformSetting` key.
- A new real-Postgres concurrency test (`platformAdmin.test.ts`) fires two competing PATCH requests concurrently (`Promise.all`, two genuinely separate `prisma.$transaction` calls, no mocking) and asserts the resulting audit chain is sequentially coherent: exactly one audit row per successful write, the first committer's row reads the true pre-race baseline, the second (final) committer's row reads the first committer's freshly committed value (not a stale duplicate of the baseline), and the final `PlatformSetting` value matches whichever write actually committed last. A second new test fires one valid and one FK-violating (forced failure) request concurrently and confirms the failing transaction's rollback never corrupts the concurrently succeeding write's committed value or audit row.
- `writePlatformAdminAuditEventInTx()`'s parameter type was narrowed to `Prisma.TransactionClient` only (the `Pick<PrismaClient, ...>` union was removed), making the "requires a transaction client" invariant a compile-time guarantee rather than a documented convention. This is a type-only change; the sole call site already passed a real `tx` and required no modification.
- `writeAuditLogInTx()` (the pre-existing, general-purpose audit helper in `utils/auditLog.ts`) was intentionally left unchanged — this correction is scoped to `platformAdminAudit.ts` only.
- Backend `tsc --noEmit` (exit 0), `npx prisma migrate status` (up to date, no new migration needed — this correction is code-only), `platformAdmin.test.ts` (51/51, including the two new concurrency tests) and `legacyConsentCorrection.test.ts` (36/36, unaffected) were all re-run against the same disposable Postgres instance. No frontend file was touched by this correction.

No production system was touched by this third correction either, and R-061 remains `OPEN` for the same reason as §7.2: this is a repository-level correctness fix to an unmerged branch, not a deployment or activation event.
