# 53 — KVKK Attachment & Imaging Lifecycle (Physical File Governance)

**Report date:** 2026-07-15 (updated same day — PR #160 review remediation pass, a second follow-up pass fixing a migration/schema drift, a third follow-up pass fixing an S3 temp-file leak and an export-reservation race condition, then a fourth follow-up pass fixing a local-copy partial-artifact leak and replacing createdAt-based staleness with a real lease/heartbeat, all found by independent review)
**Branch:** `feature/kvkk-attachment-and-imaging-lifecycle` (based off `main`)
**Status:** Implemented — awaiting deployment/operational verification. No production/browser verification has occurred. Not merged.
**Related tracker:** `docs/compliance/KVKK_COMPLIANCE_AUDIT_AND_REMEDIATION.md` — see the new remediation row added there (ID `KVKK-ATTACH-IMAGING-001`), which cross-references `KVKK-HIGH-001` (attachment encryption) and `KVKK-HIGH-003` (medical-record retention).

**PR #160 review update:** a human review round found several merge-blocking issues in the first version of this PR (unsafe blanket "deletable" classification with a live-delete endpoint not bound to any privacy-request workflow, whole-archive in-memory buffering with no size bounds, a download token carried in a URL query parameter with no true one-time atomicity, `req.user.clinicId` used directly instead of the org/branch-scoped helper on the attachments legal-hold route, no env kill-switch on the new cleanup job). All of these have been remediated; the sections below describe the corrected design, not the original one. **The live deletion-review execute endpoint has been removed entirely** (not hardened) — this PR now ships dry-run deletion-review only, see Section 4.5.

**Follow-up review update (same day):** the remediation above initially shipped the `PatientPrivacyExportArchive.status` column and its nullable-artifact-column relaxation as a *second*, separate migration, which left the first (hand-cleaned) migration's `CREATE TABLE` stale relative to `schema.prisma` — a real P0 that would have made a from-scratch `prisma migrate deploy` produce a table the export-reservation code could not actually write to. Fixed by merging both migrations back into the single original KVKK migration file and proving the merged migration matches `schema.prisma` exactly on a completely empty disposable database (`information_schema` inspection + `prisma migrate diff`, both zero-diff for every table this PR owns) — see Sections 12–13.

**Second follow-up review update (same day):** an independent review of that remediation found two further P0s: (1) `saveFileFromPath()` uploaded the export ZIP to S3 but never removed the local temp file, and the caller assumed it had been consumed — a sensitive patient ZIP could be left indefinitely under the OS temp directory, invisible to both the TTL cleanup job (which only knows about the storage key) and disk-usage monitoring; (2) `reserveGenerationSlot()` performed a `findFirst` then a `create` with no transaction-level lock, so two concurrent export requests for the same clinic could both read "no active generation" and both create a `generating` row, defeating the one-in-flight-per-clinic guarantee entirely. Both are now fixed — see Section 4.3 (reservation) and Section 10 (temp-file cleanup) — and a P1 (raw exception objects logged on export read/upload/download paths, which can embed local paths or S3 endpoints) was fixed alongside them.

**Fourth follow-up review update (same day):** an independent review of the third-round fix found two further P0s: (1) `saveFileFromPath()`'s local-mode copy-fallback streamed directly into the **final** storage path — if the write failed after partially writing data, the source temp file had already been removed but the partial, truncated final ZIP remained at its real storage key with no DB `storageKey` reference pointing at it and no TTL cleanup path (the cleanup job only knows about rows it can find via the DB, and this row never got one); (2) `reserveGenerationSlot()`'s staleness sweep treated any `queued`/`generating` row older than `EXPORT_GENERATION_STALE_MS` (10 minutes) as abandoned by `createdAt` alone — but a legitimate export can legitimately run for close to `EXPORT_MAX_TOTAL_SIZE_BYTES` (2 GB) and take longer than 10 minutes, so a slow-but-healthy export could be marked `failed` by a second request's reservation attempt while still running, and the original worker could then flip its own row back to `ready` after that happened (a lost-update race with no guard). Both are now fixed: local-mode `saveFileFromPath()` never writes directly to the final path (always via a unique `.partial-<uuid>` sibling, promoted by an atomic same-directory rename — Section 4.3/10), and export generation now carries a real lease/heartbeat (`heartbeatAt`/`leaseExpiresAt` columns, periodic renewal, and a guarded completion `updateMany`) instead of age-based staleness — see Section 4.3 and Section 12 for the disposable-Postgres proof. A CI investigation also confirmed the `windows-bridge-pr` workflow's "Windows Bridge .NET tests" job failures on this branch are pre-existing, environment-timing flakiness unrelated to this PR (which makes zero changes under `windows-bridge/`) — filed as [#161](https://github.com/MustafaBasol/DisKlinikCRM/issues/161), see Section 12.

## 1. Why this exists

Prior KVKK remediation work (`ClinicLegalProfile`, `ChannelConsentLog`, `PublicBookingNoticeEvidence`, patient JSON privacy export, patient anonymization, `deletion_review` privacy requests, `dataRetentionCleanupJob.ts`) closed gaps in structured database records and messaging metadata, but none of it touched **physical files**: `PatientAttachment` and `ImagingStudy`/`ImagingImage` rows and the bytes they point to in `fileStorage.ts` (local disk or S3-compatible object storage). `patientAnonymization.ts` redacted every other patient-linked table but never touched attachment/imaging metadata, and no attachment/imaging deletion existed for the `deletion_review` privacy-request flow. This document covers the remediation of that specific gap.

## 3. Confirmed gaps closed by this PR

1. No way to place a legal hold on an attachment or imaging study to block redaction/deletion.
2. No existence/size check on stored files without opening a full stream (needed for orphan detection).
3. No downloadable export package containing actual attachment bytes (only JSON metadata).
4. `anonymizePatientData()` never redacted attachment/imaging metadata, and had no notion of partial failure.
5. No dry-run inventory of what a "deletion review" would actually cover for attachments/imaging.
6. No narrow, auditable, idempotent live-delete path for non-clinical attachments under a confirmed deletion review.
7. No orphan-file visibility (DB row present, physical file missing) at all.

## 4. Architecture implemented

### 4.1 Storage contract extension (`server/src/services/fileStorage.ts`)

- `isSafeStorageKey(ref)` — rejects absolute paths and any `..` path segment (posix or windows separators). This is the security gate for every **new** code path added by this PR; the pre-existing absolute-path fallback (`resolveLocalPath`) is legacy-only and was never extended to any new function.
- `fileExists(ref)` — HEAD/stat existence check without opening a stream; returns `false` for any unsafe key without touching disk/S3.
- `statFile(ref)` — size metadata lookup without opening a stream; returns `null` for unsafe/missing keys.
- `buildExportStorageKey(clinicId, exportId)` — produces `exports/<clinicId>/<uuid>.zip`. `clinicId` comes from the authenticated session's resolved patient scope and `exportId` from `crypto.randomUUID()` — no user input ever feeds a path segment, so path traversal is structurally impossible, mirroring the existing `buildStorageKey` guarantee.

### 4.2 Legal hold

- `PatientAttachment.legalHold` (`Boolean @default(false)`) + `legalHoldReason` (`String?`).
- `ImagingStudy.legalHold` + `legalHoldReason`. **`ImagingImage` has no field of its own** — every image inherits its parent study's hold, checked via a join (`ImagingImage.study.legalHold`) everywhere a legal-hold check is needed (anonymization, deletion-review inventory, orphan-check reporting).
- New PATCH endpoints, both restricted to `OWNER`/`ORG_ADMIN` only (narrower than the general `PRIVACY_MANAGE_ROLES`, which also includes `CLINIC_MANAGER`):
  - `PATCH /api/patients/:patientId/attachments/:id/legal-hold` (`server/src/routes/attachments.ts`)
  - `PATCH /api/imaging/studies/:id/legal-hold` (`server/src/routes/imaging.ts`)
  - Both require `{ legalHold: boolean, reason: string }`; `reason` (min 3 chars) is now required in **both directions** — placing AND releasing a hold — per PR #160 review (releasing a hold is just as consequential as placing one: it re-opens the row to anonymization redaction).
  - `attachments.ts`'s route resolves its clinic scope via `validateAndGetClinicIdScope` (the same helper `imaging.ts`'s `findStudyInScope` already used) instead of trusting `req.user.clinicId` directly — the original version of this PR had this bug on the attachments route only; `imaging.ts` was already correct.
  - Both routes now write an `AuditLog` entry for every hold-set/hold-release event (`patient_attachment_legal_hold_set`/`_released` and `imaging_study_legal_hold_updated`), including actor id, reason, and previous/new value. The original version of this PR had no audit log call at all on the attachments route.
- **No automatic trigger sets legal hold in this PR.** The field/mechanism exists; deciding *who* may invoke it under *what* circumstances (beyond the role restriction already coded) is a remaining legal/operational decision (Section 9).
- `patientAnonymization.ts` re-reads `legalHold` fresh from the database at redaction time (via its own `select: { legalHold: true }` queries) rather than trusting any value passed into it — confirmed unchanged/correct by this review pass, no code change needed there.

### 4.3 Privacy export archive

New model `PatientPrivacyExportArchive` (`server/prisma/schema.prisma`), following the token-hash + snapshot + expiry pattern already established by `PublicBookingNoticeEvidence`, extended with a lifecycle `status` column (PR #160 review):

- Fields: `id`, `organizationId`, `clinicId`, `patientId`, `requestedByUserId`, `status` (`queued` | `generating` | `ready` | `failed`, default `ready` for the column but always explicitly set by the service), `storageKey` (nullable — points to the zip via `fileStorage.ts`, **never** serialized to any API response), `manifestJson` (`Json`, nullable), `tokenHash` (`@unique`, nullable, SHA-256 of a random download token — the raw token is returned once and never persisted), `expiresAt` (nullable, 1 hour TTL once ready), `downloadedAt` (nullable — the atomic one-time-consumption marker), `createdAt`. `storageKey`/`manifestJson`/`tokenHash`/`expiresAt` are nullable because a row is created with `status="generating"` **before** the ZIP build starts (so a concurrent request for the same clinic can be rejected with 409 instead of racing), and only populated once generation completes. (A second review pass found the migration's `CREATE TABLE` had fallen out of sync with this shape — see Section 13 for the fix and Section 12 for the disposable-DB proof.)
- **Streaming, not buffering** — `server/src/services/privacy/patientPrivacyExportPackage.ts`'s `createExportPackage()`:
  - Pipes `archiver`'s ZIP output directly into a temp file on disk (`fs.createWriteStream`), then moves/streams it into final storage via a new `fileStorage.ts` helper, `saveFileFromPath()` (local: rename, falling back to a streamed copy on cross-device rename failure; S3: a real multipart streaming `Upload` from `@aws-sdk/lib-storage`, added as a new dependency at the same major version as the existing `@aws-sdk/client-s3`). The full ZIP is never accumulated as an in-process `Buffer`/`Buffer[]`.
  - **Temp-file cleanup (P0 fix, third review round):** `saveFileFromPath()` now **always** removes the local temp file itself before returning or throwing, in every mode — S3 success, S3 failure (`try`/`finally` around `upload.done()`), local rename success (nothing left to remove), and local copy-fallback (`try`/`finally` around the streamed copy, so the source is removed whether the copy succeeded or failed). The caller no longer needs to (and does not) rely on assuming the helper consumed the path — it only nulls out its own reference as a redundant safety net. The multipart `Upload` explicitly sets `leavePartsOnError: false` (matches the library default, made explicit) so a failed multipart upload aborts and cleans up already-uploaded parts rather than leaving them orphaned/billed in the bucket. Covered by new tests exercising both local-mode paths and a mocked-S3-client success/failure pair (`kvkkAttachmentImagingLifecycle.test.ts`, tests 51–54).
  - **No partial artifact at the final path (P0 fix, fourth review round).** The local-mode copy-fallback previously streamed straight into the **final** storage path (`localPath`) — a write failure partway through left a truncated file sitting at the real storage key with no DB row pointing at it (the export row is only ever populated with `storageKey` *after* `saveFileFromPath()` returns successfully) and therefore no TTL cleanup path either. `saveFileFromPath()` now never writes to `localPath` directly: the temp file is renamed (fast path) or streamed via `stream.pipeline` (cross-device fallback) into a unique sibling `${localPath}.partial-<uuid>` path first, and only a same-directory (same-filesystem, therefore atomic) rename promotes that fully-written partial file to `localPath`. A `finally` block unconditionally removes whichever of the source temp path / partial path is still present, on both success (no-ops — both already consumed by the renames) and failure (both actually still on disk). Net effect on failure: the final storage key never exists, no `.partial-*` sibling survives, and the source temp path is gone — the DB row is the only place a caller can observe the failure, and it is marked `failed` by the caller's own catch block (see below), never left in a state pointing at a real-but-broken file. Covered by new tests: a real mid-copy failure that first flushes genuine partial bytes to the `.partial-<uuid>` path via a custom `Writable`, then fails on a later chunk, asserting none of temp/partial/final survive (`kvkkAttachmentImagingLifecycle.test.ts`, test 56); and a successful copy-fallback asserting no stray `.partial-*` sibling is left behind (test 57).
  - Enforces three bounds before reading any file bytes (via a cheap `statFile()` call): `EXPORT_MAX_FILE_COUNT = 500`, `EXPORT_MAX_FILE_SIZE_BYTES` = the existing attachment upload cap (`ATTACHMENT_MAX_FILE_SIZE_BYTES`, 10 MB, now exported from `routes/attachments.ts` and reused rather than inventing a new number), `EXPORT_MAX_TOTAL_SIZE_BYTES = 2 GB`. A file that exceeds a bound is recorded in `manifest.skippedFiles` with a stable reason code and never read; once the count or total-size bound is reached, all further attachments are skipped the same way rather than silently truncating the archive.
  - Individual attachment files (already capped at ≤10 MB by the bound above) are still read into a bounded buffer to compute their SHA-256 and append them to the archive — this is safe precisely because that size is bounded; what was eliminated is the *unbounded, whole-archive* buffering, not all buffering everywhere.
  - Manifest `missingFiles`/`skippedFiles` reasons are now stable string codes only — `file_not_found_in_storage`, `read_failed`, `size_limit_exceeded`, `count_limit_exceeded`, `total_size_limit_exceeded` — never a raw exception message (the original version interpolated `err.message` directly into a manifest field that was also returned in the API response body; that leaked internal error text to end users and audit metadata).
  - If the `PatientPrivacyExportArchive` DB update fails after the physical ZIP was already written to storage, the physical file is deleted (`deleteFile(storageKey)`) before the error propagates — no orphaned file is left behind.
  - **Per-clinic concurrency (P0 fix, third review round — now truly atomic):** the original `reserveGenerationSlot()` performed a `findFirst` (check for an active row) followed by a separate `create`, with no lock between them — under PostgreSQL's default READ COMMITTED isolation, two concurrent requests for the same clinic could both read "no active row" before either committed, so both could create a `generating` row and the one-in-flight guarantee was not actually enforced. This is now closed the same way appointment-slot booking already is (`services/appointmentRequestSafety.ts`'s `acquireAppointmentSlotLock`): `reserveGenerationSlot()` opens a `prisma.$transaction`, immediately acquires a **PostgreSQL advisory transaction lock** keyed deterministically off `clinicId` (`computeExportLockKey` — SHA-256 of a namespaced string, split into the two signed int32 values `pg_advisory_xact_lock(int4, int4)` requires), and only then, inside that same transaction: (1) sweeps any `queued`/`generating` rows for the clinic whose **lease** has expired; (2) checks for a current active row and throws `ExportGenerationInProgressError` if one exists; (3) creates exactly one new `generating` row with a fresh lease. The lock releases automatically when the transaction commits or rolls back — no in-memory mutex, so this is correct across multiple Node processes/API instances, not just within one. Proven against a real disposable Postgres database with genuinely concurrent calls to the actual exported `reserveGenerationSlot()` (not a reimplementation) via `Promise.allSettled` — see Section 12 and `scripts/verify-export-archive-lifecycle.ts`.
  - **Lease/heartbeat replaces createdAt-based staleness (P0 fix, fourth review round).** The original staleness sweep failed any `queued`/`generating` row older than `EXPORT_GENERATION_STALE_MS` (10 minutes) purely by `createdAt` age — but a legitimate export can run close to that long given the 2 GB total-size bound, so a slow-but-healthy export could be failed out from under itself by a second request's reservation attempt, and (with no guard on the completion write) the original worker could then flip its own row back to `ready` afterward — a lost-update race, not merely a false-positive failure. Two new nullable columns, `heartbeatAt`/`leaseExpiresAt`, plus a Prisma-managed `updatedAt`, replace age as the staleness signal: `reserveGenerationSlot()` stamps a fresh `heartbeatAt`/`leaseExpiresAt` (now + `EXPORT_LEASE_DURATION_MS`, still 10 minutes — but as a *lease* duration without renewal, not a hard age cap) on create; `createExportPackage()` starts a background timer (`startExportLeaseHeartbeat`, every `EXPORT_LEASE_RENEWAL_INTERVAL_MS` = 2 minutes, always `clearInterval`'d in a `finally`) that calls `renewExportLease()`, and additionally renews after every `EXPORT_LEASE_RENEWAL_FILE_BATCH` (25) attachment files processed — so a long-running, actively-progressing export keeps its lease alive indefinitely regardless of total elapsed time, while a worker that has actually died or stalled stops renewing and its lease genuinely expires. The stale sweep in `reserveGenerationSlot()` now only fails a row when `leaseExpiresAt` is null or in the past — never merely because `createdAt` is old. Completion (`generating` → `ready`) uses a **guarded `updateMany`** (`where: { id, status: 'generating', OR: [{ leaseExpiresAt: null }, { leaseExpiresAt: { gte: now } }] }`) instead of an unconditional `update` — if another process's sweep already failed the row, or the lease expired without a sweep having run yet, this matches zero rows, and the worker throws a new `ExportLeaseLostError` (mapped to HTTP 503 in `patientPrivacy.ts`) instead of returning a download token; the archive it just wrote to storage is deleted (`deleteFile(storageKey)`) before the error propagates, so no orphaned ZIP with no live DB reference is left in storage either. The catch block's failure-marking write is likewise guarded to `status: 'generating'` only, so it can never resurrect/clobber a row another process already settled. `renewExportLease()` itself is guarded to `status: 'generating'` too, so a late/racing renewal can never revive an already-failed or already-completed row. Proven against a real disposable Postgres database — see Section 12.
  - `validateExportDownloadToken()` — resolves purely by the SHA-256 hash of the client-supplied token, then cross-checks `clinicId`/`organizationId`/`patientId`/`exportId` scope, `status === 'ready'`, expiry, and whether it was already downloaded (`downloadedAt`) — returning a distinct `already_downloaded` failure code for replay attempts, separate from `not_found`/`expired`/`wrong_scope`/`not_ready`. Dependency-injectable (`client` param) for unit testing without a real DB.
  - `claimExportDownload()` — atomically marks a row downloaded via `prisma.patientPrivacyExportArchive.updateMany({ where: { id, downloadedAt: null }, data: { downloadedAt: now } })` and inspects the affected-row count: if `0`, a concurrent request already won the race and this one gets `{ claimed: false, failure: 'already_downloaded' }` rather than being served the file. This closes the concurrent-replay window that the read-only `validateExportDownloadToken` check alone could not close.
  - `cleanupExpiredExportArchives()` (dependency-injected `findExpired`/`deleteRow`/`deleteStoredFile`, mirroring the existing `dataRetentionCleanupJob.test.ts` style) — now also sweeps abandoned `queued`/`generating` rows past the staleness window, in addition to expired `ready` rows.
- Routes (`server/src/routes/patientPrivacy.ts`):
  - `POST /api/patients/:id/privacy/export-package` — builds the package, returns `{ exportId, downloadToken, expiresAt, manifest }` (raw token shown once) — **never** the storage key. Returns **409** if another export is already generating for the clinic.
  - `GET /api/patients/:id/privacy/export-package/:exportId/download` — the one-time token is now read from the **`X-Export-Download-Token` request header**, never a query parameter (query strings land in access/proxy logs and browser history — this was a real finding in PR #160 review). The route: validates the token (read-only), opens the file stream (confirming the file can actually be read — a storage-layer failure must not burn the one-time token), **then** atomically claims the download via `claimExportDownload()` (before piping to the response, closing the replay window), then streams. Three distinct audit events are written: `patient_data_export_package_download_started` (token claimed), `patient_data_export_package_downloaded` (stream completed, on the readable stream's `'end'` event), and `patient_data_export_package_download_failed` (stream error or client-disconnect/abort, detected via the response's `'close'` event firing before `'end'`) — the original version wrote a single "downloaded" audit event **before** the stream even started, so a client that disconnected mid-download was still recorded as a successful download. The raw token value is never written to any log/audit call (source-scan tested).
- Existing `POST /patients/:id/privacy/export` (JSON body) is unchanged/kept for backward compatibility.
- New dedicated cleanup job `server/src/jobs/patientPrivacyExportCleanupJob.ts` (node-cron, every 15 minutes, `withJobLock`-guarded — mirrors `publicBookingNoticeEvidenceCleanupJob.ts` exactly) deletes expired `PatientPrivacyExportArchive` rows **and** their physical zip files, regardless of `downloadedAt` — these are short-lived transfer artifacts, not audit records; the `AuditLog` entries written at download time are the durable record. Registered in `startBackgroundJobs.ts`. Now gated by a new env kill-switch, `PATIENT_PRIVACY_EXPORT_CLEANUP_ENABLED` (default enabled, `!== 'false'`) — see Section 4.7.

### 4.7 Cleanup job toggles (two, deliberately separate)

- `PATIENT_PRIVACY_EXPORT_CLEANUP_ENABLED` (new, this PR; default enabled) gates `patientPrivacyExportCleanupJob.ts` only.
- `DATA_RETENTION_CLEANUP_ENABLED` (pre-existing) gates the unrelated general `dataRetentionCleanupJob.ts`, which also has a `PlatformSetting`-backed runtime override (`getPlatformSetting('privacy.dataRetention.runtimeEnabled')`).
- These are **intentionally two separate toggles**, matching this repo's established convention of one dedicated job per feature (e.g. `publicBookingNoticeEvidenceCleanupJob.ts` is likewise independent of the general retention job). The export-archive cleanup job does **not** also have a `PlatformSetting` runtime override in this PR — an env var is considered sufficient for this low-risk, non-medical, short-TTL (1 hour) cleanup. That is a deliberate, documented scope decision for this PR, not an oversight; a `PlatformSetting` override can be added later if operational experience shows it's needed.

### 4.4 Anonymization extension (`server/src/services/privacy/patientAnonymization.ts`)

Two new steps added to `anonymizePatientData()`:

- `redactPatientAttachments()` — for every `PatientAttachment` of the patient: if `legalHold` is true, **skip entirely** (counted as `skippedLegalHold`, not redacted — legal hold preserves the item as-is for legal review, it does not merely block deletion); if `originalName === '[ANONYMIZED]'` already, skip as a no-op (idempotent re-run safety); otherwise set `originalName` to `'[ANONYMIZED]'`. `fileName`/`filePath` (already non-identifying server-generated storage keys) and the physical bytes are never touched. Each row is wrapped in its own try/catch so one failure never aborts the loop; failures are counted, not swallowed.
- `redactPatientImagingImages()` — identical semantics for `ImagingImage` rows, joined through `ImagingStudy.patientId`, using the parent study's `legalHold` (images have no field of their own).
- `AnonymizePatientResult` now includes `attachmentResults`/`imagingResults` (`{ total, redacted, skippedLegalHold, failed }`) and `partialFailure: boolean` (true if either counter's `failed > 0`).
- The pre-existing `alreadyAnonymized` idempotent early-return path now **also** runs the attachment/imaging redaction pass (so patients anonymized before this feature shipped still get their attachments/imaging redacted on a repeat call) and reports `partialFailure` there too.
- `server/src/routes/patientPrivacy.ts`'s `POST /patients/:id/privacy/anonymize` no longer unconditionally reports `success: true` — the response now includes `partialFailure`, `attachmentResults`, `imagingResults`, and `success` is `!partialFailure`.

### 4.5 Deletion-review inventory (dry-run only — live delete REMOVED, PR #160 review)

New service `server/src/services/privacy/deletionReviewInventory.ts` — `buildDeletionReviewInventory({ clinicId, patientId, organizationId })` returns counts (never writes): structured-record counts (appointments, appointment requests, contact requests, treatment cases, payments, payment plans, tooth records), attachment counts + estimated bytes (`total`, `legalHold`, `unclassifiedRetained` = all non-legal-hold attachments), imaging counts + estimated bytes (`total`, `legalHold`, `retainedClinical` = everything, since imaging has no live-delete path at all in this PR), `blockers: string[]`, and `dryRun: true`.

New route: `GET /api/patients/:id/privacy/deletion-review` (`PRIVACY_MANAGE_ROLES`) — dry-run only, unchanged in shape except the field rename below.

**`deletableAdministrative` renamed to `unclassifiedRetained` (PR #160 review).** The original field name implied every non-legal-hold `PatientAttachment` was safe to bulk-delete. There is **no category field on `PatientAttachment` distinguishing "administrative" from anything more sensitive** — the field now explicitly means "not eligible for automated deletion in this release, pending manual/legal review," not "safe to delete."

**The live-delete endpoint has been REMOVED entirely, not hardened.** The original version of this PR shipped `POST /api/patients/:id/privacy/deletion-review/execute`, which deleted every non-legal-hold `PatientAttachment` row (DB row + physical file) with:
- no binding to a specific, approved `PatientPrivacyRequest` (a fresh `{ confirm: true, reason }` on every call, decoupled from the request-status workflow);
- no dry-run-snapshot requirement (nothing forced the caller to have actually reviewed the inventory this call would act on);
- no atomic DB+storage consistency guarantee for the delete itself.

PR #160 review flagged all three as P0. Per the reviewer's own stated fallback ("if this is too large for the PR, remove the live-delete endpoint and ship dry-run only"), the endpoint, its frontend service method (`patientPrivacyService.executeDeletionReview` in `src/services/api.ts`), and any triggering UI have been **deleted from this branch entirely** — there is no live-delete code path left anywhere in this PR. A test asserts the route now 404s (see Section 12).

**No live-delete endpoint for imaging/clinical data exists in this PR either** — matching the pre-existing schema comment "tanısal veri hiç hard-delete edilmez" ("diagnostic data is never hard-deleted") on `ImagingStudy`.

**Follow-up required before any live deletion ships:** a future PR must add (a) a lifecycle-category enum on `PatientAttachment` that genuinely distinguishes administrative documents from anything clinically/legally sensitive, and (b) an execute endpoint that only deletes items explicitly approved via a bound `PatientPrivacyRequest` (not a bare `{ confirm, reason }`), with an atomic DB+storage delete guarantee. This is tracked as a new, explicit open item (Section 16).

### 4.6 Orphan detection (bounded, patient-scoped, dry-run)

New service `server/src/services/privacy/orphanFileInspection.ts` — `inspectOrphans({ clinicId, patientId })`:

- Queries `PatientAttachment` and `ImagingImage` for a **single patient** (bounded — not a clinic-wide sweep), batched at 500 rows, and calls `fileExists()` per row to classify each as `dbRowPhysicalMissing` (row exists, file missing) or `activeLinkedObject` (file present — **never** touched/deleted by this module, regardless of legal-hold status, which is reported alongside the classification for visibility only).
- `markConfirmedMissing()` — the only "live" side effect available: stamps a new `storageVerifiedMissingAt DateTime?` field (added to both `PatientAttachment` and `ImagingImage`) on rows already confirmed missing by a prior `inspectOrphans()` call. There is nothing to physically delete for a `dbRowPhysicalMissing` row (the file is already gone) — this is purely an operator-visible marker, never gated by legal hold (nothing is deleted).
- New route: `GET /api/patients/:id/privacy/orphan-check` (`PRIVACY_MANAGE_ROLES`), patient-scoped only.

**Explicitly NOT implemented in this PR (documented follow-up):** reverse-orphan detection (a physical file exists in storage with no corresponding DB row). This would require an unbounded/unrestricted S3 `ListObjectsV2` or filesystem walk, which the architecture explicitly forbids for safety reasons (a clinic-prefixed, paginated `ListObjectsV2` with `prefix = "${clinicId}/"` would be the correct bounded approach for a future PR). Also not implemented: clinic-wide bulk orphan sweeps (kept patient-scoped to stay simple and bounded for this PR). Expired `PatientPrivacyExportArchive` rows are a separate, already-bounded "temporary expired object" category, fully handled by the dedicated cleanup job in Section 4.3 — not part of this inspection.

## 5. Lifecycle policy matrix

| Category | Chosen policy | Rationale |
|---|---|---|
| Non-legal-hold `PatientAttachment` (`unclassifiedRetained`) | Retain physical file indefinitely; redact metadata (`originalName`) on anonymization; **no live-delete path exists in this PR at all** | No category field exists to distinguish "purely administrative" from anything more sensitive — the original PR's blanket "deletable" classification and its live-delete endpoint were removed per PR #160 review rather than shipped unsafely. Live deletion requires a future PR (Section 16). |
| Clinical / DICOM `ImagingImage`/`ImagingStudy` (non-legal-hold) | Retain physical file; metadata (`originalName`) **is** redacted on anonymization; the physical file is never deleted by any path in this PR | Matches the existing schema comment that diagnostic data is never hard-deleted; retention period itself is an outstanding legal decision (Section 16) |
| `PatientPrivacyExportArchive` | Short-lived (1 hour TTL); physically deleted by a dedicated cron job regardless of download status; bounded size/count on generation | It is a transfer artifact, not a record — the `AuditLog` download-completed event is the durable record |
| Legal-hold item (attachment or imaging study) | Blocks **both** anonymization metadata redaction **and** any deletion path entirely; requires a reason (min 3 chars) to place **and** to release; both directions are audited | Preserves the item exactly as-is for legal review — legal hold is stronger than "block deletion only" |

## 6. Export behavior

See Section 4.3. Summary: `POST .../export-package` → streamed-to-temp-file zip with `data.json` + `attachments/*` (bounded: ≤500 files, ≤10 MB/file, ≤2 GB total) + `manifest.json`; manifest fields are `exportVersion`, `patientId`, `clinicId`, `generatedAt`, `includedFiles` (`{ attachmentId, label, sha256, sizeBytes }`), `missingFiles`/`skippedFiles` (`{ attachmentId, reason }`, reason is always a stable code, never raw exception text). `GET .../download` requires the one-time token via the `X-Export-Download-Token` header, atomically consumes it before streaming, and never returns a storage key/path in any response body (verified by a source-scan unit test, see Section 10). Returns 409 if a generation is already in flight for the clinic.

## 7. Anonymization behavior

See Section 4.4. Partial-failure semantics: the route response is `{ success: !partialFailure, partialFailure, attachmentResults, imagingResults, ... }` — a caller (UI or API integration) checking only `success` will correctly see `false` when any per-row redaction failed, rather than the previous unconditional `success: true`. `legalHold` is re-read fresh from the DB at redaction time (confirmed, no change needed — Section 4.2).

## 8. Deletion-review behavior

See Section 4.5. **Dry-run only in this PR — there is no live-delete endpoint at all** (the original narrow execute endpoint was removed per PR #160 review, not hardened). This is a deliberate, documented limitation pending the legal/architecture decisions in Section 16, not an oversight.

## 9. Orphan cleanup behavior

See Section 4.6. Patient-scoped, dry-run classification plus an optional missing-file marker stamp; no bulk/clinic-wide sweep; no reverse (bucket-listing) orphan detection.

## 10. Security controls

- Every new/extended route re-derives `clinicId` from an org-scoped patient lookup (`resolvePatient()`) or the shared `validateAndGetClinicIdScope`/`findStudyInScope` helpers before doing anything else — no client-supplied `clinicId` is ever trusted. (The attachments legal-hold route previously used `req.user.clinicId` directly; fixed in this revision to match `imaging.ts`'s existing correct pattern.)
- Export download token is carried via the `X-Export-Download-Token` header (never a query parameter), validated purely by its SHA-256 hash, then cross-checked against `clinicId`/`organizationId`/`patientId`/`exportId`/`status`/`downloadedAt` — a forged/guessed `exportId` with someone else's real token fails as `not_found`; a token replayed against a different clinic/org/patient fails as `wrong_scope`; a second use of an already-consumed token fails as the distinct `already_downloaded` code, enforced atomically via `updateMany`'s affected-row count, not just a prior read.
- No route added or modified in this PR ever serializes `filePath`/`storageKey` to a JSON response, and manifest/API error text is always a stable reason code, never a raw exception message (spot-checked by hand across `patientPrivacy.ts`, `attachments.ts`, `imaging.ts`, and covered by automated source-scan unit tests).
- Legal-hold PATCH endpoints are restricted to `OWNER`/`ORG_ADMIN` — narrower than `PRIVACY_MANAGE_ROLES` used elsewhere in this file (which also includes `CLINIC_MANAGER`) — and now require a reason (min 3 chars) in both directions, with an audit log entry for both.
- `isSafeStorageKey()` rejects absolute paths and `..` segments for every new storage-lookup code path; the legacy absolute-path fallback in `fileStorage.ts` was not extended to any new function.
- **Safe logging (P1 fix, third review round).** Export read/upload/download error paths (`patientPrivacyExportPackage.ts`'s attachment-read and cleanup failure logs; `patientPrivacy.ts`'s export-package create/download/audit-log-failure/stream-error catch blocks) previously logged the raw caught error object or `err.message` directly — Node's own filesystem errors (e.g. `ENOENT`) embed the absolute local path in `.message`, and an S3 SDK error can embed the endpoint/bucket. These now go through a new `server/src/utils/safeError.ts` (`safeErrorFields(err)`), which logs only a stable `errorName`/`errorCode` pair plus non-sensitive identifiers already safe to log elsewhere in this codebase (`exportId`, `attachmentId`) — never the temp path, the full storage key, patient name/phone/email, the raw token, or the raw exception object.

## 11. Failure handling

- Per-row try/catch everywhere multiple objects are processed (attachment/imaging redaction) — one failure never aborts the batch.
- Export package generation treats a missing/unreadable/oversized attachment file as a manifest `missingFiles`/`skippedFiles` entry with a stable reason code, not a fatal error for the whole export; if the physical ZIP was written but the DB update fails, the physical file is deleted (no orphan).
- Export download: the file stream is confirmed openable before the one-time token is claimed, so a storage-layer failure never burns the token for a download that never happened; a mid-stream error or client abort is audited as a distinct failed/interrupted event, never conflated with a successful download.
- Export cleanup job logs and continues past a single row's delete failure (dependency-injected test asserts this: `count` still reflects only the successful deletions), and also sweeps abandoned `generating`/`queued` rows past the staleness window so they never permanently block new exports for a clinic.

## 14. Deployment / rollback notes

- **Deploy:** `npm install` (picks up the new `@aws-sdk/lib-storage` dependency), then standard `prisma migrate deploy` against the target database (applies the single KVKK migration), then deploy the updated server/frontend build. The cron job (`patientPrivacyExportCleanupJob.ts`) starts automatically via `startBackgroundJobs.ts` unless `PATIENT_PRIVACY_EXPORT_CLEANUP_ENABLED=false` is set.
- **Rollback — corrected (third review round).** An earlier revision of this document incorrectly implied `prisma migrate resolve --rolled-back` performs or is part of a schema rollback. **It does not.** `migrate resolve --rolled-back` only edits Prisma's own `_prisma_migrations` bookkeeping table to mark a migration as not-applied — it does not touch the actual `PatientAttachment`/`ImagingStudy`/`ImagingImage`/`PatientPrivacyExportArchive` tables/columns at all. Using it against a database where the migration's DDL has actually run (which is the normal case, since this migration is purely additive) leaves the real schema in the "new" state while Prisma's bookkeeping says otherwise — a dangerous mismatch that would make a subsequent `migrate deploy` attempt to re-run DDL that already exists. **`migrate resolve --rolled-back` must never be used alone after this migration has been applied.**
  - **Normal rollback (the expected path, since the schema is purely additive):** revert the application-code commit(s) (routes/services/frontend) while **leaving the additive schema migration in place**. The new columns/table are inert if the reverted code never reads/writes them — there is no need to touch the database at all for a routine revert.
  - **Destructive schema rollback (only if the columns/table must actually be physically removed — not expected, and not part of a routine revert):** requires (a) an explicit, hand-written reverse-SQL migration (`ALTER TABLE ... DROP COLUMN`, `DROP TABLE "PatientPrivacyExportArchive"`), reviewed like any other schema change; (b) a scheduled maintenance window, since this drops real columns/a real table; (c) a fresh, verified backup taken immediately before running it; (d) post-rollback verification (`information_schema` inspection confirming the columns/table are actually gone, plus a smoke test that the reverted application code still starts and serves traffic correctly without them). No other table has a required (non-nullable) foreign key into `PatientPrivacyExportArchive`, so a `DROP TABLE` is structurally low-risk in isolation — but the maintenance-window/backup/verification steps above still apply.

Safe commands for the normal (expected) rollback path:
```
git revert <application-commit-sha>      # revert the code; do NOT touch migrations or run migrate resolve
```

Commands for a destructive schema rollback (maintenance window + backup + reverse SQL required first, see above — do not run migrate resolve on its own):
```
# 1. Take a verified backup of the target database.
# 2. Apply a hand-written, reviewed reverse migration, e.g.:
#      ALTER TABLE "PatientAttachment" DROP COLUMN "legalHold", DROP COLUMN "legalHoldReason", DROP COLUMN "storageVerifiedMissingAt";
#      ALTER TABLE "ImagingStudy" DROP COLUMN "legalHold", DROP COLUMN "legalHoldReason";
#      ALTER TABLE "ImagingImage" DROP COLUMN "storageVerifiedMissingAt";
#      DROP TABLE "PatientPrivacyExportArchive";
# 3. Only after the reverse SQL has been applied and verified, reconcile Prisma's bookkeeping:
cd server && npx prisma migrate resolve --rolled-back 20260715145843_add_kvkk_attachment_imaging_lifecycle
# 4. Re-run information_schema verification (Section 12) to confirm the columns/table are actually gone.
```

## 16. Remaining legal/product decisions (explicitly not made by this PR)

1. **Clinical image retention period** — how long `ImagingImage`/`ImagingStudy` physical files must/may be retained. Not decided; current behavior is indefinite retention (no automatic deletion of imaging in any code path).
2. **DICOM retention period** — same question specifically for DICOM-sourced studies (`studyInstanceUid`/`sopInstanceUid` populated), which may carry separate regulatory retention requirements from manually-uploaded images.
3. **Hard-deletion-of-medical-records policy** — whether/when medical records (including imaging) may ever be hard-deleted at all, and under what legal process. This PR deliberately implements **no** live-delete path for imaging/clinical data.
4. **Backup deletion expectations** — this PR only affects primary-database/primary-storage state; backup retention/purge policy for deleted attachments or anonymized metadata is out of scope and undecided.
5. **Legal-hold trigger conditions/authority** — who (beyond the coded `OWNER`/`ORG_ADMIN` role restriction) has the legal authority to place a hold, under what circumstances (litigation, regulatory inquiry, etc.), and whether any external/legal-team notification should accompany it. This PR only implements the *mechanism* (the field + a role-gated PATCH endpoint + audit log); no automatic trigger, workflow, or notification exists.
6. **NEW — lifecycle-category enum + workflow-bound execute endpoint (follow-up from PR #160 review).** Live deletion of `PatientAttachment` rows requires a future PR that adds (a) a category enum genuinely distinguishing administrative documents from anything clinically/legally sensitive, and (b) an execute endpoint bound to an explicit, approved `PatientPrivacyRequest` (not a bare `{ confirm, reason }`), with an atomic DB+storage delete guarantee. Until that ships, `unclassifiedRetained` attachments are retained indefinitely with no automated deletion path.
